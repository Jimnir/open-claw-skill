## 1. 核心概念速查

### OpenClaw 是什么

> 开源、自部署的 AI Agent 框架。通过 Gateway 连接 20+ 消息平台，让 LLM 不只是聊天，还能执行任务。

### 一句话说清

| 概念 | 一句话 |
|------|--------|
| **Skills** | 给 AI 的操作手册，定义「什么场景 + 怎么做」 |
| **Cron** | 定时调度：「每晚9点执行」 |
| **Tasks** | 后台流水账：「刚才的任务跑成功没」 |
| **Hooks** | 事件触发：「当 /new 时自动保存对话」 |
| **Standing Orders** | 常设规则：「永远存在于 AGENTS.md 的指令」 |

## 2. 技能创建流程

### 完整 6 步

```
构思 → 写 SKILL.md → 保存文件 → 创建提案 → 检查 → 应用生效
 ①          ②           ③           ④         ⑤       ⑥
```

### 详细说明

**① 构思：** 回答三个问题
- 什么场景触发这个技能？（用户说什么话时调用）
- AI 具体要做什么？（列出步骤）
- 输出长什么样？

**② 写 SKILL.md：** 结构如下

```markdown
---
name: "技能名字"
description: "一句话描述触发条件"
---

# 技能标题

## 模式1：XXX
步骤1...
步骤2...

## 模式2：YYY
步骤1...
```

**③ 保存文件：** 写入临时文件

```bash
cat > /tmp/技能名字.md << 'EOF'
粘贴你的内容
EOF
```

**④ 创建提案：**

```bash
openclaw skills workshop propose-create \
  --name "技能名字" \
  --description "描述" \
  --proposal /tmp/技能名字.md
```

**⑤ 检查提案：**（返回一个 ID，如 `xxx-20260621-xxxxxx`）

```bash
openclaw skills workshop inspect 那个ID
```

**⑥ 应用生效：**

```bash
openclaw skills workshop apply 那个ID
```

### Markdown 标题规则

```
#  一级标题     → 技能名称
## 二级标题     → 模式/大章节
### 三级标题    → 子条目
正文直接写     → 不加 #
```

## 2.5 Claude Code + CCR 配置指南（2026-06-25 新增）

### 一句话

> Claude Code + claude-code-router(CCR) = 国内免翻墙使用 Claude Code 的体验，底层走 DeepSeek V4

### 原理

```
你 → Claude Code CLI → CCR代理(localhost:3456) → 协议转换 → DeepSeek V4 API
      (编辑器体验)       (请求中转)                (请求/响应适配)     (实际算力)
```

### 安装步骤

```bash
# 1. 切 npm 国内镜像源（加速下载）
npm config set registry https://registry.npmmirror.com/

# 2. 安装 Claude Code
npm install -g @anthropic-ai/claude-code

# 3. 安装 claude-code-router
npm install -g @musistudio/claude-code-router
```

### 配置 CCR（连接 DeepSeek V4）

配置文件：`~/.claude-code-router/config.json`

```json
{
  "LOG": false,
  "HOST": "127.0.0.1",
  "API_TIMEOUT_MS": 120000,
  "Providers": [
    {
      "name": "deepseek",
      "api_base_url": "https://api.deepseek.com/chat/completions",
      "api_key": "你的DeepSeek API Key",
      "models": ["deepseek-v4-flash", "deepseek-chat", "deepseek-reasoner"],
      "transformer": { "use": ["deepseek"] }
    }
  ],
  "Router": {
    "default": "deepseek,deepseek-v4-flash",
    "think": "deepseek,deepseek-v4-flash"
  }
}
```

### 使用方式

| 模式 | 命令 | 说明 |
|------|------|------|
| **交互模式** | `ccr code` | 进入对话界面，来回聊天，像在终端里和我对话 |
| **非交互模式** | `ccr code -p "问题"` | 问一句就退出，适合脚本调用 |
| **允许写文件** | `ccr code --dangerously-skip-permissions` | 跳过文件读写权限确认 |
| **启动代理** | `ccr start` | 启动 CCR 代理服务 |
| **查看状态** | `ccr status` | 查看代理是否在运行 |
| **停止代理** | `ccr stop` | 停止服务 |

> `-p` 是 print 模式的缩写，表示"问完就出结果然后退出"。不加 `-p` 进入聊天界面，可以来回对话。

### Claude Code 实战技巧

| 场景 | 命令 | 说明 |
|------|------|------|
| 代码审查 | `ccr code -p "审查项目代码"` | 一次性审查全部代码 |
| 解释代码 | `ccr code -p "解释这个文件"` | 读文件并解释 |
| 改代码 | `ccr code -p "加个 --version 参数"` | 自动修改并保存 |
| 写测试 | `ccr code -p "写 pytest 单元测试"` | 自动创建测试文件 |
| 交互修改 | `ccr code` | 进入聊天界面来回改 |

用 Claude Code 对 `/home/h/chepai/` 项目做了全面改进：

## 3.5 多 Agent 配置指南（2026-06-25 新增）

### 一句话

OpenClaw 支持配置多个 Agent，每个 Agent 有不同的人格、不同的工作区、不同的技能集。

### 当前配置

当前有 4 个 Agent：

| Agent | ID | 角色 | 工作区 |
|-------|-----|------|--------|
| **姬子 📚** | `main` | 私教老师（默认） | `workspace/` |
| **代码助手 💻** | `coder` | 编程助手 | `workspaces/coder/` |
| **面试官 🎯** | `interviewer` | 模拟面试 | `workspaces/interviewer/` |
| **百科全书 📖** | `encyclopedia` | 解答奇怪小问题 | `workspaces/encyclopedia/` |

### 配置结构

在 `openclaw.json` 的 `agents` 段配置：

```json5
{
  "agents": {
    "defaults": {
      "workspace": "/home/h/.openclaw/workspace",
      "model": { "primary": "deepseek/deepseek-v4-flash" }
    },
    "list": [
      { "id": "main", "name": "姬子 📚", "default": true },
      { "id": "coder", "name": "代码助手 💻", "workspace": "..." },
      { "id": "interviewer", "name": "面试官 🎯", "workspace": "..." }
    ]
  }
}
```

### Agent 人格从哪里来？

Agent 的人格来源于其工作区的 **SOUL.md** 文件：

```
workspaces/                    ← 所有 Agent 工作区
├── coder/
│   ├── SOUL.md                ← 这里定义代码助手的人格
│   ├── AGENTS.md
│   └── TOOLS.md
├── interviewer/
│   ├── SOUL.md                ← 这里定义面试官的人格
│   └── ...
└── encyclopedia/
    └── SOUL.md                ← 这里定义百科全书的人格
```

### 在网页界面使用

打开 `http://127.0.0.1:18789/` → 左上角可以切换 Agent。

### 注意事项

- `default: true` 的 Agent 是默认的（当前就是姬子）
- 每个 Agent 可以有不同的技能、模型、工作区
- 修改配置后需要 `openclaw gateway restart`

## 4. Skills 编写指南

### 你是我的技能库（3个）

| 技能 | 描述 | 模式 |
|------|------|------|
| **python-tutor** 🐍 | Python 作业辅导 | 调试、写代码、讲知识点、检查作业 |
| **docker-deploy** 🐳 | Docker 部署助手 | 生成 Dockerfile、生成 docker-compose、检查配置 |
| **interview-tutor** 🎤 | 面试辅导 | 分析问题 → 搜实例 → 输出 3 个回答 + 2 个实例 |

### SKILL.md 模板

```markdown
---
name: "你的技能名"
description: "告诉AI什么时候触发这个技能"
---

# 技能标题

技能说明

## 模式1：XXX
1. 步骤一
2. 步骤二

## 模式2：YYY
1. 步骤一
2. 步骤二
```

### 注意
- `description` 要写清楚触发条件
- 步骤要写具体，比如「使用 web_search 搜索」
- 输出格式要明确

### 技能库目录结构（2026-06-23 更新）

现在技能按业务场景分成了 3 个文件夹：

```
skills/
├── devops/                   ← DevOps 场景
│   └── docker-deploy/        → 容器化部署
│       ├── SKILL.md          （给AI看的操作手册）
│       └── README.md         （给人看的使用说明）
│
├── interview-prep/           ← 面试准备场景
│   └── interview-tutor/      → 面试题拆解
│
└── tutoring/                 ← 教学辅导场景
    └── python-tutor/         → Python 辅导
```

**怎么分类？** 按业务场景分，而不是随便放。以后加新技能先问自己：
- 这个技能用在哪个场景？
- 放进对应的分类文件夹
- 没有合适的分类 → 新建一个

**OpenClaw 支持嵌套目录**，`skills/devops/docker-deploy/SKILL.md` 这种写法照样能被识别。

## 4.5 技能库管理

### 怎么写 README（使用说明）

README = 使用说明书，放在项目文件夹里，让人一看就知道是干什么的。

**README 通用框架：**
```
1️⃣ 技能简介 —— 一句话说清楚这是干啥的
2️⃣ 使用场景 —— 什么时候该调用它
3️⃣ 触发方式 —— 用什么关键词能激活
4️⃣ 工作模式 —— 每种模式下它会怎么工作
5️⃣ 使用建议 —— 怎么用效果最好
6️⃣ 依赖 —— 需要哪些外部工具
```

**README 用在什么地方？**
- GitHub 仓库首页自动展示（仓库根目录放一个总的 README）
- 技能文件夹里方便自己以后回顾
- 别人想用你的技能，先看 README 决定要不要

### 技能分类整理步骤

```bash
# 1. 创建分类文件夹
mkdir -p 分类名1 分类名2 分类名3

# 2. 把技能文件夹移进去
mv 技能名 分类名/

# 3. 提交到 git
git add 分类文件夹路径
git rm 旧路径/SKILL.md
git commit -m "说明"
git push
```

**关键：**
- 先 `openclaw skills list` 确认技能还在
- git 会自动识别为 rename，不会丢历史

### 技能库使用规范（README）写法

放在 GitHub 仓库根目录的 `README.md`，别人一进仓库就能看到。

**包含哪些内容：**
1. 仓库简介 —— 一句话说明这是干什么的
2. 技能列表 —— 分类列出所有技能及其作用
3. 安装指南 —— 别人怎么把技能装到自己环境
4. 贡献指南 —— 自己以后怎么加新技能（步骤清单）
5. 命名规范 —— 文件夹命名规则、文件格式要求

### 命名规则
- 技能文件夹：全小写英文 + 短横线连接（如 `docker-deploy`）
- 描述说明：用中文写
- 每个技能包含 SKILL.md（AI操作手册）+ README.md（使用说明）

### ⚙️ Cron 定时任务

Cron 可以让 OpenClaw 在指定时间自动执行任务。

**Cron 表达式格式：**
```
分 时 日 月 周
0  22 *  *  *   ← 每天 22:00 执行
0  23 *  *  *   ← 每天 23:00 执行
```

**关键：**
- Cron 任务需要 Gateway 后台在运行才能触发
- 电脑关机后 Gateway 停止，Cron 不会执行
- 如果开机时间晚于 Cron 设定的时间，当天的任务就错过了
- 建议 Cron 时间设在你确定已经开机的时刻（如晚上 11 点）

**查看 Cron 任务：**
```bash
openclaw cron list
```

**强制触发一次（不等时间）：**
```bash
openclaw cron run --force <任务ID>
```

### 🎯 实战：AI 行业日报工作流

搭了一个每天 23:00 自动收集 AI 新闻的 Cron 工作流：

```
每天 23:00
    ↓
Cron 触发 → 派一个子代理
              ↓
          搜中文AI新闻 + 搜英文AI新闻
              ↓
          整理成日报格式（头条/动态/工具/数据）
              ↓
          写入 memory/ai-daily-YYYY-MM-DD.md
              ↓
          推送到你面前
```

**你可能用到的场景：**
- 每日面试日报
- 技能库健康检查
- 定时备份 GitHub

**注意：**
- 日报存档在 `memory/ai-daily-YYYY-MM-DD.md`
- 同时也会存一份到 Windows D 盘：`D:\ai-daily\YYYY-MM-DD.md`（路径：`/mnt/d/ai-daily/`）
- 每天一份，方便回顾

## 5. Tasks

### 一句话

> Tasks 是 OpenClaw 的后台工作流水账，不是调度器。

Cron 决定「什么时候干」，Tasks 记录「干得怎么样」。

### 会不会产生 Task？

| 来源 | 产生 Task？ |
|------|:-----------:|
| Cron 执行 | ✅ 每次记 |
| Subagent 子代理 | ✅ |
| ACP 远程调用 | ✅ |
| CLI 操作 | ✅ |
| 普通聊天 | ❌ |
| Heartbeat 心跳 | ❌ |

### 生命周期

```
queued → running → succeeded / failed / timed_out / cancelled / lost
排队      运行中      成功      失败      超时       取消     丢失
```

### UUID 是什么

```
7c57c39d-6a91-45cf-8f98-bfee5b9ef262
8-4-4-4-12 共32个十六进制字符 → 全球唯一身份证
```

### 查看命令

```bash
openclaw tasks list              # 列出所有
openclaw tasks list --runtime cron  # 只看 Cron 的
openclaw tasks show <任务ID>     # 看单条详情
openclaw tasks audit             # 检查异常
```

### 关键字段

| 字段 | 含义 |
|------|------|
| taskId | UUID 标识 |
| runtime | 来源（cron/subagent/cli/acp） |
| status | 当前状态 |
| deliveryStatus | 结果是否投递回主会话 |
| cleanupAfter | 7天后自动删除 |

## 6. Triggers

### Hooks（事件钩子）

> 当 XXX 发生时，自动执行 YYY

**系统自带的 5 个 Hook：**

| Hook | 触发时机 | 作用 |
|------|---------|------|
| 💾 session-memory | `/new` 或 `/reset` | 自动保存对话到 memory/ |
| 🚀 boot-md | Gateway 启动 | 执行 BOOT.md |
| 📝 command-logger | 任何斜杠命令 | 记录到日志文件 |
| 🧹 compaction-notifier | 会话压缩前后 | 发通知提示 |
| 📎 bootstrap-extra-files | 会话启动 | 额外注入文件 |

**常用命令：**

```bash
openclaw hooks list              # 查看所有
openclaw hooks info <名字>      # 查看详情
openclaw hooks enable <名字>    # 启用
openclaw hooks disable <名字>   # 禁用
```

### Standing Orders（常设指令）

> 不需要每次说，Agent 一直记得的规则

**你的已经有的：**
- `AGENTS.md` — 行为规范
- `SOUL.md` — 性格设定

**写法示例：**

```markdown
## Program: 每日学习提醒

**Authority:** 每晚9点检查学习进度
**Trigger:** Cron 每晚9点执行
**Approval gate:** 无需审批
```

### 三者关系图

```
Cron          Hooks        Standing Orders
(定时)       (事件触发)     (持续规则)
  │             │               │
  └──────┬──────┘               │
         │                      │
     Agent 执行                 │
         │                      │
         ▼                      │
     Tasks 记录 ◄───────────────┘
```

## 7. Git + GitHub

### 第一次推送

```bash
# 1. 配置身份
cd ~/.openclaw/workspace
git config user.name "你的用户名"
git config user.email "你的邮箱"

# 2. 创建 .gitignore（过滤敏感文件）
cat > .gitignore << 'EOF'
.openclaw/
memory/
openclaw-workspace-state.json
*.log
.env
# 按需添加其他文件
EOF

# 3. 关联远程仓库
git remote add origin https://github.com/用户名/仓库名.git

# 4. GitHub 授权
sudo apt install gh -y          # 安装 GitHub CLI
gh auth login                   # 登录（选 HTTPS + 浏览器授权）

# 5. 添加、提交、推送
git add 你要提交的文件或目录
git commit -m "提交说明"
git push -u origin master
```

### 后续推送（不用再重复前面的步骤）

```bash
git add 新文件或修改的文件
git commit -m "更新说明"
git push
```

### 提示
- 先 `git status` 看看哪些文件变了
- `.gitignore` 里列的文件不会被提交
- 敏感的配置/密码永远不要上传

## 8. 问答环节

### Q: `#` 是干什么的？

Markdown 标题符号。`#` 越多、字越小、层级越深。

```
# 一级    技能名称
## 二级   模式/章节
### 三级  子条目
正文     直接写
```

### Q: `EOF` 是干什么的？

End Of File 的缩写，是 `cat` 命令的结束标记，不是命令。

```
cat > 文件 << 'EOF'
内容...
EOF     ← 在这里告诉电脑：内容输入结束
```

也可以用别的词：`END`、`STOP` 都行。

### Q: 忘记命令怎么办？

```bash
openclaw --help              # 看所有命令
openclaw skills --help       # 看技能相关命令
openclaw skills workshop --help  # 看提案相关命令
```

或者在聊天窗口直接问我 😄

### Q: 那些英文数字组合是什么？

UUID（通用唯一标识符），格式：`8-4-4-4-12` 共 32 位十六进制。

```bash
7c57c39d-6a91-45cf-8f98-bfee5b9ef262
```

全球唯一，不会重复，系统内部用来关联记录。

### 📌 知识点

**npm 国内镜像的本质：** 合法缓存。淘宝/华为/清华等机构在国内搭服务器定期同步 npm 官方仓库，免费提供加速下载。它们靠这个引流到自家云服务。

**CCR 代理原理：** Claude Code 以为自己连 Anthropic API，CCR 在中间拦截请求，把 Claude 的协议格式转成 OpenAI 兼容格式后发给其他模型（如 DeepSeek）。

**模板匹配的局限：** 教学演示够用，实际场景下精度受限（字体差异、透视变形、光照干扰）。
